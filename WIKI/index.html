Dunit OS Green Tea

Небольшая экспериментальная ОС с Rust kernel, userspace ELF и собственным GUI.

Dunit OS Green Tea разрабатывается как практическая x86_64-система для экспериментов с загрузкой через Limine, изоляцией userspace, syscall ABI, VFS, процессной моделью и современным графическим окружением.

Rust no_std kernel C/NASM HAL Limine boot VFS/MemFS Userspace ELF Green Tea GUI

Что такое Dunit

Dunit OS - это hobby operating system, но цель проекта не ограничивается демонстрационным boot screen. Система постепенно собирает реальные основы настольной ОС: memory subsystem, файловую модель, userspace runtime, process table, syscall interface, terminal mode и отдельный GUI mode.

Название текущего системного образа - Green Tea. В интерфейсе это выражено темной темой, зелеными акцентами, плавными панелями и визуальной стилистикой, которая должна ощущаться как собственная среда Dunit, а не копия чужой оболочки.

Архитектура

Проект разделен на несколько уровней. Низкий уровень отвечает за вход в 64-bit mode, дескрипторные таблицы, прерывания и syscall entry. Ядро на Rust управляет памятью, процессами, VFS и ELF loading. Userspace-приложения собираются как отдельные Rust ELF binaries и используют `libdunit` для системных вызовов.

Boot и HAL

Limine передает framebuffer, memory map и управление ядру. HAL на C/NASM поднимает GDT, IDT, interrupt path и syscall entry.

Kernel services

Ядро содержит PMM/VMM, heap, VFS/MemFS, initrd extraction, process table, scheduler foundation и ELF loader.

Userspace

Системные приложения лежат в `userspace/system_apps/` и запускаются через runtime exec ABI с argv/env.

Текущий статус

Подсистема Состояние
Boot Limine boot flow работает, доступны terminal и GUI boot entries.
Memory Есть PMM, VMM, heap и базовая AddressSpace foundation.
Filesystem VFS и MemFS работают как foundation; initrd распаковывает встроенные приложения и assets.
Userspace Работают ELF loading, argv/env ABI, stdout/stderr, path lookup и recoverable user faults.
Processes Есть process table, parent/child foundation, honest spawn/wait semantics и cooperative scheduler foundation.
GUI Kernel GUI loop имеет wallpaper, back buffer, dirty cursor redraw, dock, windows, widgets и Green Tea visual theme.

Syscall ABI

Dunit использует инструкцию `syscall` на x86_64. Номер системного вызова передается в `RAX`, аргументы - в `RDI`, `RSI`, `RDX`, `R10`, `R8`, `R9`, результат возвращается через `RAX`. Отрицательные значения используются для ошибок.

Уже есть базовые вызовы для `read`, `write`, `open`, `close`, framebuffer access, keyboard/mouse input, `spawn`, `wait`, `getpid`, `kill`, `sleep`, `getcwd`, `chdir` и cooperative `yield`. Нереализованные или foundation-only возможности должны возвращать честные статусы вроде `EAGAIN`, `ENOSYS` или `EOPNOTSUPP`, без fake-success поведения.

Сборка и запуск

Минимальные зависимости: `gcc`, `nasm`, `make`, `xorriso`, `ld.lld`, `rustup`, `rust-src` и `qemu-system-x86_64`.

git clone https://github.com/susopki/dunit-os.git
cd dunit-os
make iso
qemu-system-x86_64 -boot d,menu=on -cdrom build/microkernel.iso -m 512M \
  -serial stdio -display sdl,grab-mod=lctrl-lalt,show-cursor=off

Для автоматической проверки используется `build_and_run_multipass.py`. В проекте он служит единой точкой сборки, запуска QEMU, остановки VM и анализа serial log.

Roadmap

Ближайшая цель

Довести process model до реального cooperative context switching: валидные runnable process records, CPU context ownership, save/restore path и честное переключение по `yield`.

Среднесрочно

Расширить userspace terminal, улучшить stdin, отделить GUI services от kernel-only loop, сделать устойчивую модель окон и подготовить реальные userspace-приложения для desktop mode.

Позже

Disk-backed filesystem, network stack, ACPI, SMP и более полноценная desktop-среда с отдельными системными сервисами.
